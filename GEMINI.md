# GEMINI.md — Project Instructions for @dotclaude/dotclaude

This file provides architectural context, development conventions, and operational guidelines for `@dotclaude/dotclaude`.

## Project Overview

`@dotclaude/dotclaude` is an opinionated toolkit for Claude Code. It serves a dual purpose:

- **Toolkit Library:** A curated collection of skills, slash commands, and subagents for Claude Code.
- **Governance CLI:** A spec-driven-development (SDD) governance tool for repos, providing bootstrapping, drift detection, and PR-time validation gates.

### Core Technologies

- **Runtime:** Node.js (>= 20, ES Modules)
- **Language:** JavaScript (No TypeScript; JSDoc for typing)
- **Testing:** Vitest (JS), BATS (Shell scripts)
- **Linting:** Prettier, markdownlint-cli2, ShellCheck
- **Architecture:** Monorepo dual-persona (plugin + dotfiles)

---

## Directory Structure

- `plugins/dotclaude/`: Core logic and CLI binaries.
  - `src/`: Public Node API and internal logic.
  - `bin/`: CLI entrypoints.
  - `scripts/`: Internal utility scripts.
  - `templates/`: Scaffolding templates for `dotclaude init`.
- `skills/`: Reusable workflows and specialists (e.g., `/aws-specialist`, `/handoff`).
- `commands/`: Explicit slash-command prompt templates (e.g., `/ground-first`).
- `agents/`: Specialized subagents.
- `docs/`: Comprehensive documentation, ADRs, and specs.
- `schemas/`: JSON schemas for skills, commands, and configs.
- `index/`: Inventory of skills and commands generated by `dotclaude index`.

## Building and Running

### Key Commands

- **Install Dependencies:** `npm install`
- **Build Plugin:** `npm run build-plugin`
- **Run Tests:** `npm test` (or `npm run test:watch` / `npm run coverage`)
- **Linting:** `npm run lint` (Checks Prettier, markdownlint, and JSDoc coverage)
- **Formatting:** `npm run format`
- **Shell Check:** `npm run shellcheck`
- **Dogfooding:** `npm run dogfood` (Validates the repo using its own CLI tools)

## Development Conventions

### "No-TypeScript" Policy (ADR-0002)

- Do **not** use TypeScript. Use JSDoc for type definitions and documentation.
- Maintain zero runtime dependencies outside of necessary ones (like `ajv` and `js-yaml`).
- All new files must be ES modules (`.mjs` or `.js` with `"type": "module"`).

### Testing Practices

- **TDD:** Write a failing test first for any bug fix or feature.
- **Verification:** Run the full test suite (`npm test`) before proposing a PR.
- **Pre-existing Failures:** Verify if a failure is pre-existing by stashing changes: `git stash && npm test; git stash pop`.

### PR and Governance

- **Protected Paths:** Changes to `CLAUDE.md`, `README.md`, `.github/workflows/**`, `.claude/**`, and `plugins/dotclaude/` are protected.
- **PR Requirements:**
  - `## Summary`: 1-3 bullets.
  - `## Test plan`: Bulleted checklist.
  - `## Spec ID`: Must be an H2 heading followed by the Spec ID (e.g., `Spec ID: dotclaude-core`).
- **Worktrees:** Use git worktrees for non-trivial changes (`.claude/worktrees/<slug>/`).

### Shell Scripting

- Use `bash` (not `zsh`).
- Use `$CLAUDE_PROJECT_DIR` in hooks and scripts.

## Governance & SDD

The project follows Spec-Driven Development.

- `dotclaude-validate-specs`: Audits spec contracts.
- `dotclaude-check-spec-coverage`: PR gate ensuring protected paths are spec-backed.
- `dotclaude-check-instruction-drift`: Detects stale entries in instruction files.
